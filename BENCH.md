# Three-Axis Reasoning Benchmark

Pure-reasoning prompts on three orthogonal axes:

- **Logical** ‚Äî ZebraLogic (logic-grid CSPs, `grid_mode`)
- **Probabilistic** ‚Äî CLadder (causal queries, Pearl's hierarchy)
- **Knowledge** ‚Äî GPQA-Diamond (graduate-level science MC)

JSON output via `response_format` (strict `json_schema`, fallback `json_object`). Schema-validated; `no_answer` is scored as wrong (no carve-out). Per-example JSONL streams to `data/bench_runs/{model}__{bench}.jsonl` and rerunning resumes.

Two harnesses (interchangeable, same JSONL schema):
- `moe_infer/helpers/bench_axes.py` ‚Äî async-openai + pydantic
- `experiment/bench.go` ‚Äî sashabaranov/go-openai, sequential verify-loop with reasoning replay (`<think>` wrap), per-row `extras.{mode,exit_reason,validate_code,attempts}`

## Method

Canonical baseline subset (id + prompt + gold per example) frozen at `data/bench_subsets/{zebralogic,cladder,gpqa_diamond}.jsonl`. Generated by `bench_axes.{ZebraLogic,CLadder,GPQADiamond}.load()` with `--n 200, no --sample-seed`. All future runs that target the BENCH.md headline numbers must match these IDs.

| Axis | Dataset | Subset | Schema | Scoring |
|---|---|---|---|---|
| Logical | `allenai/ZebraLogicBench-private` `grid_mode` | `3*3` + `4*4` (80, easy tier of 1000) | `{header, rows}` | puzzle-wise + cell-wise |
| Probabilistic | `causalNLP/cladder` `full_v1.5_default` | rungs 2 + 3, N=200 | `{answer: "yes"\|"no"}` | exact match |
| Knowledge | `Idavidrein/gpqa` `gpqa_diamond` | full 198 | `{answer: "A"\|"B"\|"C"\|"D"}` | exact letter (shuffled options) |

## Notes

- **oMLX does not enforce `response_format`.** Direct probe confirms `json_schema` strict mode is accepted but unconstrained server-side ‚Äî output discipline is intrinsic to the model. Our Backend still sends strict + `additionalProperties: false`/required-all (for any future server that honors it).
- **Retry loop is a free win on bracket/shape bugs.** Verify-loop replays reasoning into `<think>‚Ä¶</think>` so the model patches malformed JSON instead of re-deriving the answer (which can flip a correct row to wrong at temp=1.0). On VibeThinker-3B mixed_4_6, ZL first-attempt accuracy is 53.2% (25/47) vs 70.2% (33/47) final ‚Äî retry lift +17pp, ~50% recovery on the 16 first-attempt failures, 7 hit max_retries.
- **Rung 3 (counterfactual) collapse** is CLadder's diagnostic signal ‚Äî strong models lose 10-30pp from rung 2 (DeepSeek 90 ‚Üí 77).
- **`max_tokens=32k+` recovered the literature GPQA number** (83.8% vs 88.1% reported), zero truncations; the prior 16k run truncated 24%.

## How we elicit the structured answer

Per-bench `system_prompt` instructs the model to "respond with a single JSON object" matching the schema; `response_format = strict json_schema` (where the server honors it) or `json_object`. JSON-in-message-content, not tool calls.

**Verify-loop with reasoning replay.** Validate returns `(parsed, hint, err)` ‚Äî `err` is a sentinel (`ErrJSONParse`/`ErrShape`/`ErrEnum`) for analytics; `hint` is the LLM-facing message appended on failure. The retry conversation includes the prior assistant turn wrapped as `<think>{reasoning_content}</think>{content}` so Qwen3-family models can continue from their own CoT rather than re-derive (re-derivation at temp=1.0 routinely flips correct answers to wrong).

**Why not tool calls.** Tried `tools=[{submit_answer: schema}]` with `tool_choice` forcing the call. Works for instruct models (Qwen3-4B emits a clean tool call in ~5s). **Does NOT work for RL-on-text reasoning models** ‚Äî VibeThinker-3B reads the tool definitions, reasons about them in `content`, then never emits `<tool_call>` and stops at `finish_reason=length`. The base (Qwen2.5-Coder-3B) supports tool use and the chat template still carries the slots, but the reasoning RL stage trained the emission away. Expected to apply to other reasoning-RL'd models.
